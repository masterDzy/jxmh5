## 六、异常处理

```typescript
const ERROR_HANDLING = {
  // 1. 单用户失败不影响整批
  onUserError: (userId, error) => {
    logError(`用户 ${userId} 运势生成失败: ${error.message}`);
    continue; // 继续处理下一位
    writeFailedUser(userId, error); // 记失败表，稍后重试
  },

  // 2. API 超时 / 429
  onRateLimit: async (retryCount) => {
    if (retryCount < 3) {
      await sleep(Math.pow(2, retryCount) * 1000); // 指数退避
      return retry();
    }
    throw new Error("Rate limit exceeded after 3 retries");
  },

  // 3. 批量完成后的汇总
  onBatchComplete: (results) => {
    const success = results.filter(r => !r.error).length;
    const failed = results.filter(r => r.error).length;
    return `本月运势生成完成：成功 ${success} 条，失败 ${failed} 条`;
  }
};
```

## 七、待决策项

| 序号 | 待决策项 | 选项 |
|------|---------|------|
| 1 | **触发粒度** | 只流月 / 流年+流月 / 流年+流月+流旬 全做 |
| 2 | **八字精度** | 属相only / 属相+年 / 完整八字（年月日时） |
| 3 | **用户输入** | 用户主动触发 / 后台静默生成 / 两者都有 |
| 4 | **推送方式** | 前端轮询拉取 / 飞书通知 / 微信通知 / 全部 |
| 5 | **失败重试** | 失败用户是否需要单独 cron 重跑 |
| 6 | **历史存档** | 是否保留历史运势记录，还是只留最新 |

## 八、技术选型结论

| 环节 | 方案 | 理由 |
|------|------|------|
| 预测生成 | OpenClaw sub-agent | 预测是高智力环节，需要强思维链；MiniMax-M2 已内化奇门/十神体系 |
| 定时触发 | OpenClaw cron | 与生成端同层，环境统一，无额外运维成本 |
| 通知路由 | Hermes（未来扩展） | 预测服务成熟后，Hermes 负责分发到各渠道 |
| 工作流平台 | 暂不引入 | 引入 n8n/Dify 增加运维复杂度，且不节省智力成本 |

> Dify / n8n 适合"已有预测服务 API"时的编排层，不适合作为预测生成端。
